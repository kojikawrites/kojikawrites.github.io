# Example Site

A minimal example site demonstrating the blog platform capabilities. This README serves as the reference documentation for site configuration.

## What's Included

This example site demonstrates:

### Content
- ✅ **Blog post** (`2025-11-17-welcome-to-example-site.mdx`) - Published post with today's date
- ✅ **Draft post** (`_drafts/2099-12-25-future-post.mdx`) - Future-dated draft post
- ✅ **About page** (`content/pagecontent/about.mdx`) - Static page with site information
- ✅ **Test page** (`content/pagecontent/test.mdx`) - Page showcasing markdown features

### Components
- ✅ **ExampleComponent.astro** - Minimal reusable Astro component with props and slots

### Configuration
- ✅ **site.yaml** - Site configuration (see [Site Configuration](#site-configuration-siteyaml) below)
- ✅ **.env.example** - Site-specific environment variables template
- ✅ **logo-map.json** - Auto-generated logo mapping for date-based logos

## Usage

To use this example site:

1. **Set as active site** in root `.env`:
   ```env
   SITE_CODE=example.com
   ```

2. **Copy the environment file**:
   ```bash
   cp src/.sites/example.com/.env.example src/.sites/example.com/.env
   ```

3. **Start development server**:
   ```bash
   npm run dev
   ```

3. **Visit**: https://localhost:4321

## Site Configuration (site.yaml)

The `config/site.yaml` file contains all site-specific configuration. Some sections are auto-generated by build scripts, while others are manually configured.

### Configuration Sections

#### `main` - General Site Settings

```yaml
main:
  logo:
    alt: "Site Name"                    # Alt text for logo
    dynamic_dir: /src/.sites/.../logos/dynamic  # Directory for dynamic logo variants (optional)
```

**Dynamic Logos (Advanced Feature):**

The `dynamic_dir` setting enables date-based logo variants (e.g., holiday logos, special occasions). When configured, the build script scans this directory and generates a `state/logo-map.json` file.

**Directory Structure:**
```
images/logos/dynamic/
├── hiive-logo-dark.svg          # Default dark theme logo
├── hiive-logo-light.svg         # Default light theme logo
├── alt.txt                      # Alt text for default logos (optional)
├── 2025-07-04/                  # Exact date: July 4th, 2025 only
│   ├── logo-dark.svg
│   ├── logo-light.svg
│   └── alt.txt                  # "Happy Independence Day!"
├── xxxx-12-25/                  # Recurring: Every December 25th
│   ├── logo-dark.svg
│   ├── logo-light.svg
│   └── alt.txt
└── xxxx-06-xx/                  # Recurring: All of June (every year)
    ├── logo-dark.svg
    ├── logo-light.svg
    └── alt.txt                  # "Happy Pride Month!"
```

**Date Pattern Format:** `YYYY-MM-DD` with wildcards:
- `xxxx` = any year
- `xx` = any month or day

**Pattern Priority (highest to lowest):**
1. `YYYY-MM-DD` - Exact date (e.g., `2025-07-04`)
2. `xxxx-MM-DD` - Recurring exact day (e.g., `xxxx-12-25` for Christmas)
3. `YYYY-xx-DD` - Specific day every month in a year
4. `xxxx-xx-DD` - Specific day every month
5. `YYYY-MM-xx` - All days of one month in a year
6. `xxxx-MM-xx` - All days of a month yearly (e.g., `xxxx-06-xx` for Pride Month)
7. `YYYY-xx-xx` - All days of a year
8. `default` - Fallback for all other days

**File Naming Convention:**
- Logo files must end with `-dark.svg` or `-light.svg` to indicate theme
- Alt text is read from `alt.txt` in the same directory (falls back to `main.logo.alt`)

**Testing:** Set `PUBLIC_LOGO_TEST_DATE=YYYY-MM-DD` in `.env` to test date-specific logos in development.

#### `build` - Build Configuration

```yaml
build:
  exclude_from_production:              # Directories to exclude from production builds
    - api
    - admin
```

These directories will:
- Be excluded from frontmatter extraction
- Be removed from the production build output
- Remain fully functional in development mode

#### `navbar` - Navigation Bar

```yaml
navbar:
  logo:
    dark: /src/.sites/.../logo-dark.svg   # Logo for dark theme
    light: /src/.sites/.../logo-light.svg # Logo for light theme
    alt: "Site Name"                       # Alt text for logo
  left:                                    # AUTO-GENERATED - Left navigation items
    - href: /blog
      label: blog
  right:                                   # AUTO-GENERATED - Right navigation items
    - href: /about
      label: About
  breadcrumbs:
    include: true                          # Enable/disable breadcrumbs
    valid_breadcrumbs:                     # Base breadcrumb mappings (auto-augmented at runtime)
      - href: '[slug]'
        label: '[slug]'
```

**Auto-generated fields:**
- `navbar.left` and `navbar.right` are regenerated by `generateNavMenu.ts` when:
  - Content pages in `content/pagecontent/` have `showInMenu: true` in frontmatter
  - System menu items exist in `state/system-menu-items.json`
- `navbar.breadcrumbs.valid_breadcrumbs` is dynamically augmented at runtime with page titles

#### `blog` - Blog Settings

```yaml
blog:
  pagination_size: 15                    # Posts per page on blog listing
  path: blog                             # URL path for blog (e.g., /blog/...)
  small_mode: card                       # Mobile layout: "card" or "list"
  large_mode: card                       # Desktop layout: "card" or "list"
  default_author: authorname             # Default author for posts without one
```

**Layout modes:**
- `card` - Card-style with background, border, thumbnail side-by-side
- `list` - Compact list-style layout

You can mix modes: `small_mode: list` + `large_mode: card` shows list on mobile, cards on desktop.

#### `footer` - Footer Content

```yaml
footer:
  content:                               # Array of footer lines
    - "Company Name and logo are trademarks."
    - "All content © 2005-[CURRENT_YEAR] by Company."  # [CURRENT_YEAR] is replaced at runtime
```

#### `inject_components` - Component Injection

Sites can inject custom components into every page without modifying the core framework. This is useful for site-wide backgrounds, overlays, analytics, or other persistent UI elements.

```yaml
inject_components:
  - component: starfield/StarfieldBackground
    props:
      opacity: 0.7
      zIndex: -999
  - component: analytics/Tracker
    props:
      trackingId: "UA-123456"
```

**Configuration:**
- `component`: Path to the component relative to `src/.sites/[site-name]/components/`
- `props`: Object of props to pass to the component (optional)

**How it works:**
- Components are dynamically imported using `import.meta.glob`
- Injected into `BaseLayout.astro` inside the `<main>` tag
- Components can persist across page navigations when using View Transitions (with `transition:persist`)
- Each site can define its own set of injected components independently

**Example use cases:**
- Animated backgrounds (WebGL shaders, particle systems)
- Site-wide modals or notification systems
- Analytics and tracking scripts
- Floating action buttons or navigation aids
- Theme-specific decorative elements

### Page Frontmatter for Menu Generation

To add a content page to the navigation menu, include these fields in the page's frontmatter:

```yaml
---
title: About Us
showInMenu: true           # Required to appear in menu
menuLabel: About           # Optional: override display label (defaults to title)
menuPosition: right        # "left", "right", or "none"
menuOrder: 100             # Lower numbers appear first
---
```

### System Menu Items

The `state/system-menu-items.json` file defines menu items for built-in pages (blog listing, tags, categories). This file is edited via Keystatic CMS under **System Menu Items** singleton.

```json
{
  "items": [
    {
      "href": "/blog",
      "label": "blog",
      "position": "left",
      "order": 1
    },
    {
      "href": "/tags",
      "label": "tags",
      "position": "left",
      "order": 2
    },
    {
      "href": "/categories",
      "label": "categories",
      "position": "left",
      "order": 3
    }
  ]
}
```

**Fields:**
- `href` - URL path for the menu item
- `label` - Display text in the navigation
- `position` - `"left"` or `"right"` side of navbar
- `order` - Sort order (lower numbers appear first)

**Note:** The blog listing href should match your `blog.path` setting in site.yaml. Common values are `/blog` or `/posts` - you can use any path you prefer.

### Menu Configuration via Keystatic

Navigation menus are managed through Keystatic CMS in two ways:

1. **System Menu Items** (`/edit` → System Menu Items singleton)
   - For built-in pages: blog listing, tags, categories
   - Add/remove/reorder items for framework-provided routes

2. **Page Menu Properties** (`/edit` → Pages → edit any page)
   - Each content page has menu settings in its editor:
     - **Show in Navigation Menu** - Toggle visibility
     - **Menu Label** - Override display text (defaults to page title)
     - **Menu Position** - Left or right side of navbar
     - **Menu Order** - Sort order (lower = first)

The `generateNavMenu.ts` script combines both sources to build the final `navbar.left` and `navbar.right` arrays in site.yaml.
